reagent is an MCP server, not just a proxy. Claude Code connects to it via stdio transport
declared in `.mcp.json`. reagent then connects outward to any downstream MCP servers you
configure in `.reagent/gateway.yaml`.

Instead of listing your filesystem server, GitHub server, and other tools directly in `.mcp.json`
(which would bypass reagent's middleware), declare them in `gateway.yaml`. Every proxied call
goes through the same 12-layer middleware chain as native tool calls — policy, redaction, audit,
rate-limiting, and the kill switch all apply.

## How it works

1. Claude Code spawns `reagent serve` on session start via `.mcp.json`
2. reagent reads `.reagent/gateway.yaml` and connects to each downstream server
3. It discovers their tools via MCP `tools/list`
4. Each tool is re-registered with a namespace prefix (`servername__toolname`)
5. Native first-party tools are registered through the same middleware chain
6. Every tool call — native or proxied — passes through the 12-layer middleware chain
7. reagent listens on stdio for incoming MCP requests from Claude Code

## `.mcp.json` (written by `reagent init`)

```json
{
  "mcpServers": {
    "reagent": {
      "type": "stdio",
      "command": "node",
      "args": ["node_modules/@bookedsolid/reagent/dist/cli/index.js", "serve"]
    }
  }
}
```

The direct `node` path works across npm, yarn, pnpm, and bun. Legacy projects using the
`npx reagent serve` pattern can migrate by running `reagent upgrade`.

Do not add downstream servers directly to `.mcp.json`. Add them to `.reagent/gateway.yaml`
so they are proxied through reagent's middleware.

## Configuration

Create or edit `.reagent/gateway.yaml`:

```yaml
version: '1'
servers:
  discord-ops:
    command: node
    args:
      - /path/to/discord-ops/dist/index.js
    env:
      DISCORD_BOT_TOKEN: '${DISCORD_BOT_TOKEN}'
    tool_overrides:
      get_messages:
        tier: read
      send_message:
        tier: write
      purge_messages:
        tier: destructive
      delete_channel:
        tier: destructive
        blocked: true

gateway:
  max_result_size_kb: 512
```

### Environment variable resolution

Use `${VAR_NAME}` syntax in env values. reagent resolves them from `process.env` at startup.
Missing env vars produce a warning and resolve to empty string.

### Tool overrides

Each downstream tool can be assigned a tier (`read`, `write`, `destructive`) and optionally
blocked entirely. Overrides cannot lower a tool's tier below its static or convention-based
classification — the override is ignored with a warning if attempted.

## Tool namespacing

Downstream tools are namespaced as `servername__toolname` to prevent collisions:

```
discord-ops__send_message
discord-ops__get_messages
filesystem__read_file
filesystem__write_file
```

## Multiple downstream servers

```yaml
version: '1'
servers:
  discord-ops:
    command: node
    args: [/path/to/discord-ops/dist/index.js]

  filesystem:
    command: npx
    args: [-y, '@modelcontextprotocol/server-filesystem', '/allowed/path']

  github:
    command: npx
    args: [-y, '@modelcontextprotocol/server-github']
    env:
      GITHUB_PERSONAL_ACCESS_TOKEN: '${GITHUB_TOKEN}'
    calls_per_minute: 120

gateway:
  max_result_size_kb: 512
```

All tools from all servers are aggregated and served over reagent's single stdio connection
to Claude Code.

## Rate limiting and circuit breaking

Per-server rate limits and concurrency limits are enforced by the middleware chain:

```yaml
postgres:
  command: npx
  args: ['@modelcontextprotocol/server-postgres', '${DATABASE_URL}']
  calls_per_minute: 60
  max_concurrent_calls: 5
  tool_overrides:
    execute_query:
      tier: destructive
```

The circuit breaker tracks consecutive downstream failures and opens after a configurable
threshold, denying calls to that server until it resets.
