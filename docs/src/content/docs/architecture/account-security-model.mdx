This document describes the security properties of reagent's multi-credential management feature — how tokens are stored, what threats the system protects against, what risks are explicitly accepted, and how the account system integrates with reagent's existing security infrastructure.

## Threat model

### What we protect against

| Threat | Mitigation |
|---|---|
| Token written to disk (file) | Config loader rejects YAML containing `sk-ant-` patterns; secret-scanner hook blocks writes containing OAuth token patterns |
| Token committed to git | Secret scanner hook runs on every file write/edit; gitleaks pre-commit hook provides secondary coverage |
| Token leaked to downstream MCP servers | `buildEnv()` in client-manager strips `CLAUDE_CODE_OAUTH_TOKEN` and `CLAUDE_CODE_OAUTH_REFRESH_TOKEN` from child process environments |
| Token leaked to AI model context | Redact middleware scans tool arguments and results for secret patterns before they reach the model |
| Token read by another OS user | macOS Keychain entries are protected by the login keychain; accessible only to the authenticated user |
| Token persisted in shell history | `reagent account env` outputs export commands for `eval`, not the token directly; `rswitch` captures and evals in one step |
| Stale/expired credential used unknowingly | `reagent account check --all` reports expiry status; `rotate` re-authenticates |

### Accepted risks

These risks are inherent to the environment model and are explicitly accepted:

| Risk | Rationale |
|---|---|
| Token visible via `env` or `/proc/*/environ` on the local machine | Any process running as your user can read your environment. This is a property of Unix, not a flaw in reagent. The token is equivalent to what Claude Code itself places in the environment when launched. |
| Token visible in `ps` output if passed as a CLI argument | reagent never passes tokens as CLI arguments — they are always set as environment variables. |
| Keychain access by any process running as the user | macOS Keychain's login keychain is unlocked while the user is logged in. Any process running as that user can read entries. This is standard macOS security behavior. |
| Token intercepted by a compromised shell plugin | If your shell is compromised (malicious zsh plugin, modified `.zshrc`), the attacker can read environment variables. This is outside reagent's threat boundary. |
| Memory-resident tokens in the shell process | The token exists in the shell's memory as an environment variable. Memory-scraping attacks are outside scope. |

## Keychain storage details

### Entry structure

Each account creates one Keychain entry:

| Property | Value | Example |
|---|---|---|
| Keychain | Login keychain (default) | `login.keychain-db` |
| Kind | Generic password | — |
| Service | `reagent-<account-name>` | `reagent-personal` |
| Account | `reagent` (constant) | `reagent` |
| Data | JSON blob | `{"accessToken":"...","refreshToken":"...","expiresAt":"..."}` |

The service name is namespaced with `reagent-` to avoid collisions with other applications. The account field is a constant `reagent` across all entries, allowing the `security` CLI to target entries precisely with both `-s` (service) and `-a` (account) flags.

### Credential shape

The stored JSON blob conforms to the `AccountCredential` interface:

```typescript
interface AccountCredential {
  accessToken: string;        // The OAuth access token (sk-ant-oat01-...)
  refreshToken?: string;      // Optional refresh token for token renewal
  expiresAt?: string | number; // Expiry timestamp (ISO 8601 string or Unix ms)
  scopes?: string[];          // OAuth scopes granted
  subscriptionType?: string;  // Plan type (e.g., "claude_max")
  rateLimitTier?: string;     // Rate limit tier for the subscription
}
```

When reading from Claude Code's own keychain entry (`Claude Code-credentials`), the fields are normalized from Claude Code's internal format (`oauth_token` to `accessToken`, `refresh_token` to `refreshToken`, `expiry` to `expiresAt`).

When activated via `CLAUDE_CODE_OAUTH_TOKEN`, Claude Code ignores `refreshToken`, `subscriptionType`, and `rateLimitTier`. These fields are stored for reagent's own health checks and display, not consumed by Claude Code.

### Keychain operations

All keychain operations use the macOS `security` CLI via `execFileSync`:

| Operation | Command |
|---|---|
| Write | `security add-generic-password -s <service> -a reagent -w <data> -U` |
| Read | `security find-generic-password -s <service> -a reagent -w` |
| Delete | `security delete-generic-password -s <service> -a reagent` |
| Exists | `security find-generic-password -s <service> -a reagent` (exit code check) |

The `-U` flag on write allows updating an existing entry. Before writing, the code attempts to delete the existing entry first (ignoring errors if it does not exist) to avoid Keychain prompts about overwriting.

### Encryption

Keychain entries are encrypted at rest using the login keychain's encryption key, which is derived from the user's macOS login password. The keychain is unlocked when the user logs in and locked when they log out or the screen locks (depending on system settings).

reagent does not implement any additional encryption layer — it relies entirely on macOS Keychain's built-in encryption.

## Token lifecycle

### Creation

1. User runs `reagent account add <name>`
2. The current Claude Code keychain credential is backed up (read from `Claude Code-credentials` service)
3. `claude auth login` runs, opening a browser for Anthropic's OAuth flow
4. On success, Claude Code writes the new credential to its own keychain entry (`Claude Code-credentials`)
5. reagent reads the new credential from Claude Code's entry and normalizes the field names
6. reagent writes the credential to a new keychain entry under `reagent-<name>`
7. The original Claude Code credential is restored from the backup

This "borrow and restore" pattern ensures that Claude Code's own credential is not overwritten by the registration process.

### Activation

1. User runs `rswitch <name>` or `eval "$(reagent account env <name>)"`
2. reagent reads the credential from `reagent-<name>` in Keychain
3. Shell `export` commands are output for `CLAUDE_CODE_OAUTH_TOKEN` and `REAGENT_ACCOUNT`
4. The shell evaluates these commands, setting the variables in the current process
5. Any `claude` process started from this shell inherits the token via standard Unix environment inheritance

### Rotation

1. User runs `reagent account rotate <name>`
2. Same backup-login-capture-restore flow as creation
3. The existing keychain entry for `reagent-<name>` is overwritten with the new credential
4. The `accounts.yaml` entry is unchanged (only the keychain data changes)

<Aside type="tip">
  Rotate proactively before tokens expire. Use `reagent account check --all` to monitor expiry dates across all accounts.
</Aside>

### Deactivation

1. User runs `rswitch --clear` or `eval "$(reagent account env --clear)"`
2. `unset` commands remove `CLAUDE_CODE_OAUTH_TOKEN` and `REAGENT_ACCOUNT`
3. Claude Code falls back to its keychain default (priority 5 in the auth chain)

### Deletion

1. User runs `reagent account remove <name>`
2. The keychain entry `reagent-<name>` is deleted via `security delete-generic-password`
3. The account is removed from `~/.reagent/accounts.yaml`
4. If the account was active in the current shell, the user is prompted to clear the environment

## Env var exposure surface

When an account is active, the token is present in:

| Location | Visibility | Notes |
|---|---|---|
| Shell environment (`$CLAUDE_CODE_OAUTH_TOKEN`) | Current shell process and children | Standard Unix env inheritance |
| `/proc/<pid>/environ` (Linux) | Same-user processes | Not applicable on macOS; included for completeness |
| `ps eww <pid>` (macOS) | Same-user processes | Shows environment of running processes |
| Child process environment | Inherited by default | Except: reagent strips it from downstream MCP server processes |

The token is NOT present in:

| Location | Why |
|---|---|
| `~/.reagent/accounts.yaml` | Config stores keychain service name, never the token |
| `.reagent/` project directory | No account data is stored in project directories |
| Git history | Secret scanner blocks file writes containing `sk-ant-` patterns |
| Downstream MCP server processes | `buildEnv()` strips OAuth tokens before spawning child processes |
| MCP tool arguments/results | Redact middleware scans for and replaces secret patterns |

## Integration with reagent security infrastructure

### Secret scanner hook

The `secret-scanner.sh` Claude Code hook fires before every file write and edit. It detects Anthropic OAuth token patterns:

```
sk-ant-oat01-[A-Za-z0-9_-]{86}
```

This is classified as HIGH severity — any write containing this pattern is blocked (exit code 2) before the file is modified.

The hook also detects Anthropic API key patterns (`sk-ant-api03-`), providing coverage for the broader credential family.

### Redact middleware

The redact middleware in the MCP server chain scans both tool arguments (before execution) and tool results (before returning to the model) for secret patterns. OAuth tokens matching known patterns are replaced with `[REDACTED]`.

### Config-level token rejection

The `accounts.ts` config loader scans the raw YAML string for `sk-ant-` patterns before parsing. If a match is found, loading throws immediately:

```
accounts.yaml contains what appears to be an inline token (sk-ant-*).
Tokens must be stored in Keychain, never in config files.
```

This check runs on both load and save, preventing tokens from being written to `accounts.yaml` through any code path.

### Downstream token stripping

The `ClientManager.buildEnv()` function in `src/gateway/client-manager.ts` explicitly removes `CLAUDE_CODE_OAUTH_TOKEN` and `CLAUDE_CODE_OAUTH_REFRESH_TOKEN` from the environment passed to downstream MCP server child processes. This prevents billing credentials from being accessible to third-party server implementations.

## Recommendations for team environments

### Individual accounts

Each team member should register their own accounts using their own credentials. Do not share keychain entries, export tokens, or create shared accounts.

### Client engagement isolation

For client work, create a dedicated account per client engagement:

```bash
reagent account add client-acme
reagent account add client-beta
```

Use `rswitch` before starting work on a client project to ensure billing routes correctly.

### Audit trail

The `REAGENT_ACCOUNT` environment variable is set alongside the token. This value can be logged or referenced in audit entries to track which billing account was active during a session.

### Token rotation policy

Rotate credentials periodically and when team members leave:

```bash
# Check all accounts for expiry
reagent account check --all

# Rotate a specific account
reagent account rotate client-acme
```

### CI/CD environments

The `reagent account` system is designed for interactive developer workstations, not CI/CD. In CI/CD pipelines, use standard environment variable injection through your CI provider's secrets management (GitHub Actions secrets, etc.) rather than Keychain-based storage.
