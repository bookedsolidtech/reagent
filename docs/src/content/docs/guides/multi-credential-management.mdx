Claude Code bills every session to a single OAuth token. If you work across multiple accounts — personal, business, client engagements — you need a way to route billing to the right subscription without logging out and back in each time.

`reagent account` solves this. It stores each account's OAuth credential in macOS Keychain under a namespaced entry and exports the right `CLAUDE_CODE_OAUTH_TOKEN` environment variable when you switch. Claude Code sees the env var, uses it instead of its keychain default, and bills accordingly.

No API keys are involved. No tokens are shared. Each account is a legitimate Claude Max or Teams subscription owned by you, authenticated through Claude Code's standard OAuth flow.

## Prerequisites

- **macOS** with Keychain Access (the `security` CLI must be available)
- **Claude Code CLI** installed and working (`claude --version`)
- **Multiple Claude accounts** — each with an active Claude Max or Teams subscription
- **reagent** installed globally or as a project devDependency

## Quick start

<Steps>

1. **Add your first account**

   ```bash
   reagent account add personal
   ```

   This opens a browser for Claude Code's standard OAuth login. Log in with the account you want to register as "personal". The credential is captured and stored in macOS Keychain under `reagent-personal`.

2. **Add another account**

   ```bash
   reagent account add client-acme
   ```

   Same flow — log in with a different account. Your original Claude Code credential is backed up and restored automatically.

3. **Set up shell integration**

   ```bash
   reagent account setup-shell >> ~/.zshrc
   source ~/.zshrc
   ```

   This adds the `rswitch` shell function and tab completion.

4. **Switch accounts**

   ```bash
   rswitch client-acme
   ```

   Or without shell integration:

   ```bash
   eval "$(reagent account env client-acme)"
   ```

5. **Verify**

   ```bash
   reagent account whoami
   ```

   Output:

   ```
   Account:  client-acme
   Billing:  (no description)
   Token:    sk-ant-oat01...x4Fg
   Status:   Active (via CLAUDE_CODE_OAUTH_TOKEN env var)
   ```

</Steps>

## CLI reference

### `reagent account add <name>`

Registers a new account by running Claude Code's OAuth login flow and storing the resulting credential in macOS Keychain.

```bash
reagent account add clarity-house
```

**What happens:**

1. Backs up your current Claude Code keychain credential
2. Runs `claude auth login` (opens browser for OAuth)
3. Reads the new credential from Claude Code's keychain entry
4. Stores it under `reagent-<name>` in Keychain
5. Restores your original Claude Code credential
6. Adds the account to `~/.reagent/accounts.yaml`

Account names must be lowercase alphanumeric with hyphens (e.g., `clarity-house`, `personal`, `client-acme`).

If the account already exists, the command exits with an error and suggests `reagent account rotate` instead.

### `reagent account list`

Shows all registered accounts and their keychain status.

```bash
reagent account list
```

Output:

```
Registered accounts:

  * personal (active) — BST engineering
    Store: keychain ok (reagent-personal)
    client-acme — ACME Corp project
    Store: keychain ok (reagent-client-acme)

```

The `*` marker and `(active)` label indicate which account is currently set via the `REAGENT_ACCOUNT` environment variable.

### `reagent account env <name>`

Outputs shell `export` commands suitable for `eval`. This is the mechanism behind `rswitch`.

```bash
eval "$(reagent account env personal)"
```

The command reads the credential from Keychain and outputs:

```bash
export CLAUDE_CODE_OAUTH_TOKEN='sk-ant-oat01-...'
export REAGENT_ACCOUNT='personal'
```

### `reagent account env --clear`

Outputs `unset` commands to remove the account override and return to Claude Code's keychain default.

```bash
eval "$(reagent account env --clear)"
```

Outputs:

```bash
unset CLAUDE_CODE_OAUTH_TOKEN
unset REAGENT_ACCOUNT
```

### `reagent account check [--all]`

Validates token health for the active account or all accounts.

```bash
# Check active account only
reagent account check

# Check all registered accounts
reagent account check --all
```

Output:

```
Account health check:

  + personal: ok
    Token:   sk-ant-oat01...x4Fg
    Refresh: present
    Expires: 2026-05-15T00:00:00.000Z
    Plan:    claude_max (max_5)
  ! client-acme: EXPIRED
    Token:   sk-ant-oat01...r2Kp
    Refresh: present
    Expires: 2026-04-01T00:00:00.000Z (EXPIRED)
```

Status indicators: `+` means healthy, `!` means action required (missing, expired, or no token).

The `Plan:` field appears when `subscriptionType` is present in the stored credential (set at registration time via Claude Code's keychain entry).

### `reagent account verify [--all]`

Calls the Anthropic API to confirm account identity — email, display name, plan type, rate limit tier, and billing type — for the active account or all accounts.

```bash
# Verify active account
reagent account verify

# Verify all registered accounts
reagent account verify --all
```

Output:

```
Account verification (via Anthropic API):

  + personal: VERIFIED
    Email:   user@example.com
    Name:    Test User
    Plan:    Max
    Tier:    max_5
    Billing: subscription
  ! client-acme: API request failed (token may be expired)
    Run: npx @bookedsolid/reagent@latest account rotate client-acme
```

**What it does:** Sends a `GET` request to `https://api.anthropic.com/api/oauth/profile` with the stored access token as a Bearer credential. The endpoint returns the account's email, display name, organization UUID, plan type, rate limit tier, and billing type — confirming that the token is valid and routing billing to the expected account.

**Plan labels:** `claude_max` displays as `Max`, `claude_pro` as `Pro`, `claude_team` as `Team`, `claude_enterprise` as `Enterprise`.

**Use this to confirm:**
- The stored token belongs to the account you expect
- The correct billing plan is active
- The token has not expired (expired tokens return an API error)

<Aside type="caution">
  **Env var token limitation.** When a token is active via `CLAUDE_CODE_OAUTH_TOKEN` (e.g., after `rswitch`), Claude Code's usage display shows "Claude API" instead of your plan name (Max, Pro, etc.). This is a Claude Code display limitation, not a billing issue — `account verify` queries the server directly and will show the correct plan name regardless. Billing and rate limits always match your actual subscription.
</Aside>

<Aside type="note">
  **Refresh tokens and env var sessions.** Token refresh is not supported when a token is active via `CLAUDE_CODE_OAUTH_TOKEN`. When a token expires (typically after ~1 hour), reagent cannot automatically refresh it. Run `npx @bookedsolid/reagent@latest account rotate <name>` followed by `rswitch <name>` to get a fresh token.
</Aside>

### `reagent account whoami`

Shows which account is currently active and how it was activated.

```bash
reagent account whoami
```

If no account is active:

```
No reagent account active.
Using Claude Code default (keychain credential).
```

### `reagent account rotate <name>`

Re-authenticates an existing account. Use this when a token expires or you need to refresh credentials.

```bash
reagent account rotate client-acme
```

The flow is identical to `add` — backs up the current credential, runs OAuth login, stores the new credential, restores the original. The difference is that `rotate` updates an existing keychain entry rather than creating a new one.

### `reagent account remove <name>`

Deletes the keychain entry and removes the account from `accounts.yaml`.

```bash
reagent account remove client-acme
```

If the removed account is currently active in the shell, the command reminds you to clear the environment:

```
Removed "client-acme" from accounts.yaml.

Note: "client-acme" is currently active in this shell.
Run: eval "$(reagent account env --clear)"
```

### `reagent account setup-shell [--shell zsh|bash|fish]`

Prints shell integration code to stdout. Pipe it to your shell config file.

<Tabs>
<TabItem label="zsh/bash">
```bash
reagent account setup-shell >> ~/.zshrc
# or
reagent account setup-shell --shell bash >> ~/.bashrc
```
</TabItem>
<TabItem label="fish">
```bash
reagent account setup-shell --shell fish >> ~/.config/fish/config.fish
```
</TabItem>
</Tabs>

The generated code provides:

- **`rswitch <name>`** — switch to a named account
- **`rswitch --clear`** — clear the account override
- **`rswitch`** (no args) — list all accounts
- **Tab completion** — completes account names and `--clear`
- **Prompt indicator** (commented out) — shows `[account-name]` in your prompt

## Configuration

Account metadata lives in `~/.reagent/accounts.yaml`. This file is in your home directory, not in any project — accounts are global.

```yaml
version: '1'
accounts:
  personal:
    description: 'BST engineering — personal Max Plan'
    credential_store: keychain
    keychain_service: reagent-personal
    budget:
      warn_usd: 150.00
      halt_usd: 200.00
      period: monthly
  client-acme:
    description: 'ACME Corp project — client-provided token'
    credential_store: keychain
    keychain_service: reagent-client-acme
    budget:
      warn_usd: 40.00
      halt_usd: 50.00
      period: monthly
```

### Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `description` | `string` | No | Human-readable label for the account |
| `credential_store` | `'keychain'` | Yes | Storage backend (only `keychain` is supported) |
| `keychain_service` | `string` | Yes | macOS Keychain service name (auto-set to `reagent-<name>`) |
| `budget.warn_usd` | `number` | No | Advisory warning threshold |
| `budget.halt_usd` | `number` | No | Hard stop threshold |
| `budget.period` | `'monthly' \| 'weekly' \| 'daily'` | No | Budget period |

<Aside type="caution">
  The budget fields are advisory metadata only. Budget enforcement is planned for a future release.
</Aside>

### Inline token rejection

The config loader scans the raw YAML for `sk-ant-` patterns before parsing. If a token value is found inline in `accounts.yaml`, loading fails immediately with an error. Tokens must live in Keychain — never in config files.

## Shell integration

### How `rswitch` works

`rswitch` is a shell function (not a binary) because it needs to modify the current shell's environment. A subprocess cannot change its parent's environment variables.

The flow:

1. `rswitch personal` calls `reagent account env personal` as a subprocess
2. `reagent account env` reads the credential from Keychain and prints `export` commands to stdout
3. `rswitch` captures that output and runs `eval` on it in the current shell
4. The current shell now has `CLAUDE_CODE_OAUTH_TOKEN` and `REAGENT_ACCOUNT` set
5. Any `claude` process started from this shell inherits those variables

### Prompt indicator

The setup-shell output includes a commented-out prompt modification:

```bash
# Optional: show active account in prompt
# PROMPT='${REAGENT_ACCOUNT:+[\$REAGENT_ACCOUNT] }'"$PROMPT"
```

Uncomment it to see `[personal]` or `[client-acme]` in your shell prompt when an account is active.

### Scope

The account switch is scoped to the shell session where you ran `rswitch`. Other terminal tabs, other tmux panes, and other users are unaffected. Opening a new terminal starts with no account override (Claude Code's keychain default).

## How it works

### Two-layer architecture

The system has two layers:

1. **Credential storage** — macOS Keychain holds the actual OAuth tokens as JSON blobs under `reagent-<name>` service entries with account name `reagent`.

2. **Credential activation** — The `CLAUDE_CODE_OAUTH_TOKEN` environment variable tells Claude Code to use a specific token instead of its keychain default. `reagent account env` reads from layer 1 and produces the shell commands for layer 2.

### Auth precedence chain

Claude Code checks for authentication credentials in this order:

| Priority | Source | Type |
|---|---|---|
| 1 | `ANTHROPIC_AUTH_TOKEN` env var | OAuth |
| 2 | `CLAUDE_CODE_OAUTH_TOKEN` env var | OAuth |
| 3 | `CLAUDE_CODE_OAUTH_TOKEN_FILE_DESCRIPTOR` | OAuth (fd) |
| 4 | `apiKeyHelper` (external command from settings) | Varies |
| 5 | Keychain-stored OAuth (`claude auth login`) | OAuth |

reagent targets priority 2 — `CLAUDE_CODE_OAUTH_TOKEN`. This is a documented, supported environment variable. The `add` and `rotate` commands explicitly delete `ANTHROPIC_AUTH_TOKEN` and `ANTHROPIC_API_KEY` from any child environment to prevent accidental precedence conflicts.

<Aside type="caution">
  When using `CLAUDE_CODE_OAUTH_TOKEN`, Claude Code's usage display will show "Claude API" instead of your actual plan name (Max, Pro, Teams). This is a display-only issue — billing and rate limits are enforced correctly server-side.
</Aside>

### Keychain storage format

Each account's credential is stored as a JSON blob in macOS Keychain:

- **Service name:** `reagent-<account-name>` (e.g., `reagent-personal`)
- **Account name:** `reagent` (constant across all entries)
- **Data:** JSON with `accessToken`, `refreshToken`, `expiresAt`, `scopes`, `subscriptionType`, and `rateLimitTier` fields

The Keychain entry is created and read using the macOS `security` CLI:

```bash
# Write (reagent does this internally)
security add-generic-password -s reagent-personal -a reagent -w '{"accessToken":"..."}' -U

# Read (reagent does this internally)
security find-generic-password -s reagent-personal -a reagent -w
```

### Why NOT `CLAUDE_CONFIG_DIR`

An alternative approach would be to maintain separate Claude Code config directories per account using the `CLAUDE_CONFIG_DIR` environment variable. This was rejected because:

- It duplicates Claude Code's entire configuration state per account
- It creates divergent settings across accounts
- It is heavier to manage and more fragile
- `CLAUDE_CODE_OAUTH_TOKEN` is the minimal, targeted mechanism — it overrides only the auth token, leaving everything else shared

### Token stripping for downstream servers

When reagent proxies tool calls to downstream MCP servers via `gateway.yaml`, it strips `CLAUDE_CODE_OAUTH_TOKEN` and `CLAUDE_CODE_OAUTH_REFRESH_TOKEN` from the child process environment. This prevents your billing credentials from leaking to third-party MCP server processes.

```typescript
// From src/gateway/client-manager.ts
const STRIP_FROM_DOWNSTREAM = [
  'CLAUDE_CODE_OAUTH_TOKEN',
  'CLAUDE_CODE_OAUTH_REFRESH_TOKEN',
];
```

## Anthropic Terms compliance

This feature uses Claude Code's supported configuration mechanisms to manage legitimate, user-owned credentials for billing routing purposes.

### Legitimate OAuth tokens

Every token managed by `reagent account` is obtained through Claude Code's standard `claude auth login` OAuth flow — the exact same browser-based flow any Claude Code user runs. No tokens are forged, fabricated, or obtained through non-standard means.

### One user, multiple accounts

The intended use case is a single user who owns and pays for multiple Claude subscriptions — for example, a consultant with a personal Max Plan and separate Max Plans for each client engagement. This is functionally equivalent to logging out and logging back in with a different account, automated for convenience.

### No credential sharing

Tokens are stored per-user in macOS Keychain, encrypted at rest and protected by the login keychain password. They are:

- Never written to files on disk (the config file stores only the keychain service name, never the token itself)
- Never shared across OS user accounts
- Never transmitted to third parties
- Never committed to version control (the config loader rejects files containing `sk-ant-` patterns)

### No API abuse

This feature does not use the Anthropic API. It uses Claude Code (the CLI tool) with Claude Max and/or Teams subscriptions. Billing is 1:1 — one session uses one account's token and bills to that account's subscription.

### No circumvention of rate limits

Each account's rate limits, usage caps, and subscription terms apply independently. Switching accounts routes billing to a different subscription — it does not pool, share, or bypass any per-account limits. If account A is rate-limited, switching to account B does not help because account B has its own independent usage metering.

### Standard environment variable mechanism

`CLAUDE_CODE_OAUTH_TOKEN` is a documented, supported environment variable in Claude Code's authentication system. reagent uses it exactly as intended — to specify which OAuth token Claude Code should use for a given session.

### Defense in depth against token leakage

The system actively prevents tokens from leaking through multiple independent mechanisms:

- **Keychain-only storage** — tokens are never written to files; the config loader rejects YAML containing `sk-ant-` patterns
- **Downstream stripping** — `CLAUDE_CODE_OAUTH_TOKEN` is stripped from child process environments when proxying to MCP servers
- **Secret scanner hook** — the `secret-scanner.sh` Claude Code hook detects `sk-ant-oat01-` patterns (Anthropic OAuth tokens) in file writes and blocks them
- **Redact middleware** — the MCP middleware chain redacts secret patterns from tool arguments and results before they reach the model

## Troubleshooting

### "No credential found in keychain"

The keychain entry for this account is missing or corrupted. Re-authenticate:

```bash
reagent account rotate <name>
```

### "Account already exists"

You are trying to `add` an account that is already registered. To refresh its credential:

```bash
reagent account rotate <name>
```

### Token is expired

Run `reagent account check --all` to identify expired tokens by expiry date, or `reagent account verify --all` to confirm which tokens are actually invalid against the API, then rotate:

```bash
reagent account rotate <name>
```

### `rswitch` command not found

The shell function is not loaded. Either:

1. Run `reagent account setup-shell >> ~/.zshrc && source ~/.zshrc`
2. Or use the direct form: `eval "$(reagent account env <name>)"`

### Switch does not take effect in other terminals

Account switches are per-shell-session. Each terminal tab or tmux pane has its own environment. Run `rswitch` in each shell where you need the override.

### Claude Code ignores the token

Check that `ANTHROPIC_AUTH_TOKEN` is not set — it has higher precedence than `CLAUDE_CODE_OAUTH_TOKEN`:

```bash
echo $ANTHROPIC_AUTH_TOKEN
```

If it is set, unset it or ensure it is not in your shell profile.

### OAuth login opens the wrong browser profile

`claude auth login` opens your default browser. If you have multiple browser profiles, make sure the correct one is active before running `reagent account add` or `reagent account rotate`. Some users find it helpful to use a private/incognito window.

## FAQ

**Can I use this on Linux?**
Not currently. The credential storage layer uses macOS Keychain (`security` CLI). Linux support (via `libsecret`/`secret-tool`) is a planned future enhancement.

**Can multiple users share an account?**
No. Tokens are stored in the user's login keychain and scoped to that OS user. Each person should register their own accounts with their own credentials.

**What happens if I switch accounts mid-session?**
The switch takes effect for the next Claude Code process you start. A running `claude` process continues using whatever token it was started with.

**Does this work with Claude Teams?**
Yes. Any Claude account that can complete the `claude auth login` flow can be registered. This includes Max Plans and Teams plans.

**Are tokens refreshed automatically?**
No. Claude Code does not use `CLAUDE_CODE_OAUTH_REFRESH_TOKEN` for env-var-based tokens. When a token expires (typically after ~1 hour), run `npx @bookedsolid/reagent@latest account rotate <name>` followed by `rswitch <name>` to get a fresh token. Use `npx @bookedsolid/reagent@latest account check --all` to monitor expiry.

**Why does Claude Code show "Claude API" instead of "Claude Max"?**
This is a known Claude Code limitation. When a token is provided via `CLAUDE_CODE_OAUTH_TOKEN`, Claude Code hardcodes the subscription type to `null` instead of querying the server. The display is cosmetic — billing still routes to the correct account and server-side rate limits match your actual plan.

**Where is `accounts.yaml` stored?**
At `~/.reagent/accounts.yaml` — in your home directory, not in any project. Accounts are global to your user, not project-specific.

**Can I have project-specific account defaults?**
Not currently. Account selection is manual via `rswitch` or `reagent account env`. Project-level account defaults are a potential future feature.
